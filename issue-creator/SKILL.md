# Issue Creator - 会话 Issue 创建器

将当前会话的工作内容、讨论和方案自动总结并创建为 GitHub Issue。

## 触发条件

当用户说以下关键词时触发：
- "创建 issue"
- "记录到 issue"
- "总结成 issue"
- "保存会话"
- "生成 issue"
- "创建任务"

## 核心功能

1. **分析会话内容** - 提取会话中的关键信息
2. **识别方案** - 检测是否已形成具体方案
3. **生成结构化内容** - 按照模板组织 Issue 内容
4. **创建 GitHub Issue** - 使用 GitHub MCP 工具创建

## 工作流程

### Step 1: 确定目标仓库

询问用户或从上下文确定：
- 仓库 owner（默认从当前项目推断）
- 仓库名称（默认从当前项目推断）

### Step 2: 分析会话内容

提取会话中的关键信息：

| 信息类型 | 提取内容 |
|---------|---------|
| **会话主题** | 本次会话的核心目标 |
| **完成的工作** | 已完成的任务和变更 |
| **形成的决策** | 关键决策和选择 |
| **方案内容** | 如已形成方案，提取完整内容 |
| **后续工作** | 待办事项和下一步计划 |

### Step 3: 判断是否有方案

检查会话是否包含可执行的方案：

**有方案的标志**：
- 包含具体的实施步骤
- 有代码示例或配置
- 明确了文件结构和变更
- 形成了可执行的计划

**无方案的情况**：
- 仅为讨论和调研
- 问题尚未解决
- 需要进一步分析

### Step 4: 生成 Issue 内容

#### Issue 标题格式
```
{type}: {简要描述}

类型选项：
- feat: 新功能
- fix: Bug 修复
- docs: 文档更新
- refactor: 重构
- chore: 维护任务
- research: 调研
```

#### Issue 内容模板

**有方案的情况**：
```markdown
## 会话总结

{会话主题和背景简述}

## 方案内容

### 实施步骤
1. {步骤 1}
2. {步骤 2}
3. ...

### 关键变更
- {变更 1}
- {变更 2}

### 代码/配置示例
```
{代码或配置内容}
```

### 注意事项
- {注意点 1}
- {注意点 2}

## 已完成的工作
- [x] {已完成的工作 1}
- [x] {已完成的工作 2}

## 后续工作
- [ ] {待办 1}
- [ ] {待办 2}
```

**无方案的情况**：
```markdown
## 会话总结

{会话主题和背景}

## 讨论内容

### 问题/目标
{描述问题或目标}

### 已尝试的方案
- {尝试 1} - 结果：{结果}
- {尝试 2} - 结果：{结果}

### 当前状态
{当前进展和阻碍}

## 需要进一步确定
- [ ] {待确定问题 1}
- [ ] {待确定问题 2}

## 参考信息
{相关链接、代码片段等}
```

### Step 5: 创建 Issue

使用 GitHub MCP 创建 Issue：

```python
issue_write(
    method="create",
    owner="{owner}",
    repo="{repo}",
    title="{type}: {标题}",
    body="{生成的内容}",
    labels=["{标签1}", "{标签2}"]
)
```

**默认标签**：
- 有方案：`["todo", "enhancement"]` 或 `["todo", "bug"]`
- 无方案：`["discussion", "help wanted"]` 或 `["research"]`

## 使用示例

### 示例 1: 会话已形成方案

```
用户: "我们已经确定了重构方案，创建 issue 记录下来"

AI:
1. 分析会话内容，识别重构方案
2. 提取实施步骤和关键变更
3. 生成包含完整方案的 Issue 内容
4. 创建 Issue：
   - 标题："refactor: 重构用户认证模块"
   - 内容包含：实施步骤、文件变更、代码示例、注意事项
   - 标签：["todo", "refactor"]
```

### 示例 2: 会话为调研讨论

```
用户: "把这个调研讨论保存成 issue"

AI:
1. 分析会话中的调研内容
2. 识别已尝试的方案和结果
3. 生成 Issue：
   - 标题："research: 数据库选型调研"
   - 内容包含：问题描述、已尝试方案、当前状态、待确定问题
   - 标签：["research", "discussion"]
```

## 输出格式

创建完成后输出报告：

```markdown
## Issue 创建报告

### 会话分析
- **会话主题**: {主题}
- **是否形成方案**: ✅ 是 / ❌ 否
- **关键决策**: {决策数量} 个
- **后续工作**: {待办数量} 个

### Issue 详情
- **标题**: {type}: {标题}
- **链接**: [#{number}]({url})
- **标签**: {标签列表}
- **状态**: 🆕 已创建

### 内容概要
{Issue 内容的简要概述}

### 下一步
1. 查看 Issue: [#{number}]({url})
2. 根据需要调整标签和优先级
3. 分配给相关成员
```

## 注意事项

1. **隐私保护** - 避免在 Issue 中记录敏感信息（密码、密钥等）
2. **简洁明了** - Issue 内容应清晰易懂，便于他人理解
3. **可操作性** - 有方案的 Issue 应该包含可执行的步骤
4. **标签规范** - 使用统一的标签体系便于管理

## 标签建议

| 场景 | 推荐标签 |
|------|---------|
| 新功能 | `["todo", "enhancement"]` |
| Bug 修复 | `["todo", "bug"]` |
| 重构 | `["todo", "refactor"]` |
| 文档 | `["todo", "documentation"]` |
| 调研 | `["research", "discussion"]` |
| 问题讨论 | `["discussion", "help wanted"]` |
